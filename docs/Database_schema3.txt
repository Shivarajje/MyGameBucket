# Table 3 — user_games

## Purpose

The `user_games` table represents a user's personal game library.

It connects:

One User

↓

One Game

while storing only **user-specific** information.

The official game information always remains inside the Game Catalog.

---

```text id="ntbjlwm"

Primary Key
```

---


---

## Philosophy

This table stores only information that belongs to the user.

Examples:

* Hours Played
* Rating
* Status
* Started Date
* Completed Date

It intentionally does **not** store:

* Game Title
* Cover Image
* Genre
* Release Year

Those belong to the Game Catalog.

Every user may own a game only once.

Constraint:

```text id="azqlp5"

```

# Automatic Statistics

The following values are **not stored**.

They are calculated dynamically.

Examples:

Total Games

↓

COUNT()

Completed Games

↓

COUNT(status = Completed)

Total Hours

↓

SUM(hours_played)

Completion Percentage

↓

Calculated

Keeping statistics dynamic prevents synchronization problems.

# Performance Notes

The `user_games` table will become one of the largest tables in the application.

Expected workload:

* Extremely high read frequency
* Moderate write frequency

Optimization priorities:

* Fast library loading
* Fast profile rendering
* Efficient filtering
* Efficient sorting
* Efficient joins with `game_catalog`

This table should be heavily indexed on commonly queried fields.

---

# Design Notes

The `user_games` table intentionally stores **only personal gameplay information**.

Everything else is retrieved by joining with `game_catalog`.

This follows the project's **Single Source of Truth** principle and avoids unnecessary data duplication.
