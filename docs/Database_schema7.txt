# Table 10 — system_logs

## Purpose

The `system_logs` table stores important backend and system events.

Unlike `admin_logs`, these records are generated automatically by the application.

They assist with:

* Debugging
* Monitoring
* Security auditing
* Performance analysis

These logs are never visible to regular users.

---

## Primary Key

```text id="qg7rkp"
id

UUID

Primary Key
```

---

## Relationships

System logs are independent.

They generally do not require foreign key relationships.

Where useful, optional references may be stored.

# Global Indexing Strategy

Indexes should prioritize the most common queries.

High Priority

* username
* slug
* game title
* profile_id

Medium Priority

* status
* visibility

# Database-Wide Constraints

The following constraints apply across the entire schema.

Profiles

↓

One Profile per User

---

User Games

↓

One Game per User

---

Journal Entries

↓

One Entry per User per Game

---

Collections

↓

Maximum 10 per User

(Backend Enforced)

---

Collection Games

↓

No Duplicate Games

---

Settings

↓

One Settings Record per User

---

Usernames

↓

Unique

Case Insensitive

---

Game Catalog

↓

Unique Slug

Unique External API ID

---

# Cascade Delete Strategy

Deleting a user removes:

Profile

↓

Settings

↓

Library

↓

Journal Entries

↓

Collections

↓

Collection Games

The Game Catalog remains untouched.
