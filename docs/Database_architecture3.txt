# Primary Keys

Every application table uses its own UUID.

Examples:

profiles

↓

id (UUID)

game_catalog

↓

id (UUID)

user_games

↓

id (UUID)

journal_entries

↓

id (UUID)

collections

↓

id (UUID)

Using UUIDs provides:

* Global uniqueness
* Better scalability
* Native Supabase compatibility
* Easier future integrations

Composite primary keys are intentionally avoided.

Relationships are enforced using foreign keys and unique constraints.

# Unique Constraints

To maintain data integrity, several uniqueness rules are enforced.

## Usernames

Unique.

Case insensitive.

Examples:

Valid:

Shivaraj

Invalid:

shivaraj

---

## User Library

Each user can own a game only once.

Constraint:

(profile_id, game_id)

UNIQUE

---

## Journal Entries

Each user can have one Journal Entry per game.

Constraint:

(profile_id, game_id)

UNIQUE

---

## Collection Contents

A collection cannot contain duplicate games.

Constraint:

(collection_id, game_id)

UNIQUE

---

The database is the final authority.

Frontend validation exists only to improve user experience.

Every important field is validated again before storage.

# Automatic Timestamps

Every major table includes:

created_at

updated_at

These timestamps are automatically managed.

Benefits:

* Sorting
* Auditing
* Analytics
* Recently Updated sections
* Easier debugging

---

# Cascade Deletes

The database automatically removes dependent records.

Example:

Delete User

↓

Delete Profile

↓

Delete Library

↓

Delete Journal Entries

↓

Delete Collections

↓

Delete Collection Games

The shared Game Catalog is never affected.

---

# Soft Deletes

Version 1 does not use soft deletes.

Deleted records are permanently removed.

Reasons:

* Simpler schema
* Simpler queries
* Less maintenance
* Better performance

A recycle bin may be introduced in a future version.

---

# Indexing Philosophy

Indexes exist to optimize frequent queries.

Priority should be given to:

* Username lookups
* Game searches
* Library queries
* Collection lookups
* Recently updated items

Indexes should improve performance without unnecessary duplication.
