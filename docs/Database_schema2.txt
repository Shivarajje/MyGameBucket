# Table 2 — game_catalog

## Purpose

Stores one shared record for every game known to MyGameBucket.

Every user references the same game.

This eliminates duplicate game information across user libraries.

The Game Catalog acts as the local index of all games that have been searched or manually approved.

## Synchronization Rules

Game data should only be refreshed when:

* Cache expires
* Data is missing
* Administrator requests refresh
* Future scheduled synchronization jobs

Every successful synchronization updates:

```text id="5l1zaj"
last_synced_at
```

---

## Performance Notes

The Game Catalog is expected to become one of the largest tables.

Expected characteristics:

* High read frequency
* Low write frequency
* Shared across every user

Optimization priorities:

* Search speed
* Lookup by slug
* Lookup by external API ID
* Fast joins with user_games

## Design Notes

The Game Catalog is intentionally minimal.

Detailed information such as:

* Description
* Screenshots
* Videos
* Developer
* Publisher

is fetched through the GameService when needed.

This keeps the database compact while still allowing fast searches and profile rendering.
